# 玩具塔防阶段一结构设计

## 总体结构

- `ATTDGameMode`：设置默认 PlayerController，确保进入默认地图后自动出现菜单 UI。
- `ATTDMenuPlayerController`：统一管理开始界面、主界面、研究所界面和战斗 HUD 的切换。
- `UTTDResearchSubsystem`：作为研究所系统的运行时数据中心，管理图鉴数据、解锁状态、玩具盒制作队列和存档。
- `FTTDCraftingQueueModel`：纯 C++ 队列模型，负责队列容量、并行倒计时、完成状态和领取逻辑，便于自动化测试。
- `UTTDObjectPoolWorldSubsystem`：作为每个 `UWorld` 的全局对象池，统一管理池化对象的获取、释放和统计。
- `UTTDBattleWorldSubsystem`：作为首版战斗运行时中心，管理战斗启动、刷怪、建造、背包、进度和胜负。
- C++ UMG Widget：提供可运行的原型界面，同时保留 Blueprint 可替换空间。

## 数据模型

- 零件：`FTTDPartDefinition`，包含 ID、名称、说明。
- 图纸：`FTTDDiagramDefinition`，包含 ID、名称、说明、所需零件列表。
- 玩具盒：`FTTDToyBoxDefinition`，包含 ID、名称、说明、可开出零件列表、制作耗时。
- 图鉴条目：`FTTDCollectionEntry`，把图纸和玩具盒统一成 UI 可消费结构，并带解锁状态。
- 制作队列项：`FTTDCraftQueueItem`，包含队列 ID、玩具盒 ID、名称、总时长、剩余时长、完成状态。
- 存档：`UTTDSaveGame`，保存已解锁图纸、已解锁玩具盒、已解锁零件、制作队列和上次保存时间。
- 对象池配置：`FTTDObjectPoolDefinition`，包含池化 Class、初始数量、最大容量、扩容数量和是否 BeginPlay 预热。
- 战斗关卡：`FTTDBattleLevelDefinition`，包含关卡 ID、城堡、初始资源和波次列表。
- 战斗波次、敌人、建筑、玩具盒奖励和高度效果器分别由 `FTTDWaveDefinition`、`FTTDEnemyDefinition`、`FTTDBuildingDefinition`、`FTTDToyBoxRewardDefinition`、`FTTDBattleHeightEffectDefinition` 承载。

## 实现策略

- 先实现研究所需求明确的内容：开始界面、主界面、研究所、图鉴、玩具盒研发队列。
- 仓库暂做入口占位；关卡挑战已接入默认战斗启动，缺配置或缺场景承载时返回明确失败原因。
- 队列最多 5 个槽位，所有队列项同步倒计时，完成后可点击领取。
- 关闭游戏期间用上次保存时间补算离线制作进度。
- 图纸研发页签先保留页面和入口，后续等图纸合成/消耗规则明确后接入。
- UI 当前使用 C++ 原型控件，后续可在蓝图里继承或替换 WidgetClass，保持数据层不变。

## 测试与验证

- 自动化测试覆盖制作队列：
  - 空队列可加入玩具盒。
  - 队列达到容量后拒绝继续加入。
  - 多个队列项同步倒计时。
  - 倒计时归零后进入完成状态。
  - 已完成项可按队列 ID 领取并从队列移除。
- 编译目标：`ToyTowerDefenseEditor Win64 Development`。
- 自动化测试路径包括 `ToyTowerDefense.Research.CraftingQueueModel`、`ToyTowerDefense.Settings.DeveloperSettings`、`ToyTowerDefense.Messaging.GameplayMessageRouter`、`ToyTowerDefense.ObjectPool.WorldSubsystem` 和 `ToyTowerDefense.Battle.WorldSubsystem`。

# DeveloperSettings 数据配置设计

## 目标

把研究所、对象池和战斗等可配置内容收敛到自定义 `UTTDDeveloperSettings`，用 `GameplayTag -> DataTable` 映射管理数据表，避免继续把玩法配置硬编码在系统里。

## 当前配置项

- `GameplayTagDataTables`：核心配置入口，Key 为 GameplayTag，Value 为 DataTable。
- `TTD.DataTable.Research.Parts`：零件表，行结构为 `FTTDPartDefinition`。
- `TTD.DataTable.Research.Diagrams`：图纸表，行结构为 `FTTDDiagramDefinition`。
- `TTD.DataTable.Research.ToyBoxes`：玩具盒表，行结构为 `FTTDToyBoxDefinition`。
- `TTD.DataTable.ObjectPool.Definitions`：对象池表，行结构为 `FTTDObjectPoolDefinition`。
- `TTD.DataTable.Battle.Levels` / `Waves` / `Enemies` / `Buildings` / `ToyBoxRewards` / `HeightEffects`：战斗配置表，行结构分别对应 `TTDBattleTypes.h` 中的战斗 row struct。
- `CraftingQueueMaxSlots`：制作队列槽位上限。
- `SaveSlotName`：研究所存档槽名称。
- `DefaultUnlockedDiagramIds` / `DefaultUnlockedToyBoxIds` / `DefaultUnlockedPartIds`：新存档默认解锁内容。
- `DefaultBattleLevelId`：主菜单“关卡挑战”默认启动的战斗关卡 ID。

## 加载策略

- 研究所初始化时先从 `UTTDDeveloperSettings` 加载三张 DataTable。
- 如果某类 DataTable 未配置或为空，则只对该类别使用当前内置样例数据兜底。
- 对象池在 WorldSubsystem 初始化时加载对象池 DataTable，BeginPlay 时按配置预热。
- 战斗启动时加载 Battle DataTable，并在清理旧战斗前完成启动校验；校验失败会返回 `OutFailureReason`。
- DataTable 行 ID 字段可留空，系统会用 RowName 填充对应 ID。
- UI 和消息系统保持不变，数据来源切换不会影响现有调用。

# GameplayMessageRouter 接入设计

## 目标

将 Lyra 的 `GameplayMessageRouter` 作为项目消息总线，让研究所、UI、后续仓库/关卡/战斗模块通过 GameplayTag Channel 解耦通信。

## 接入方式

- 项目内新增插件：`Plugins/GameplayMessageRouter`，来源为同版本 Lyra 示例插件。
- `.uproject` 启用 `GameplayMessageRouter`。
- 游戏模块依赖新增 `GameplayTags` 与 `GameplayMessageRuntime`。
- 项目消息定义集中在 `TTDGameplayMessages`，统一声明 GameplayTag Channel 与 Payload USTRUCT。

## 当前消息通道

- `TTD.Message.Research.Queue.Changed`：制作队列发生变化或制作项完成。
- `TTD.Message.Research.Queue.Enqueued`：玩具盒加入制作队列。
- `TTD.Message.Research.Queue.Claimed`：完成的玩具盒被领取。
- `TTD.Message.Research.Collection.Changed`：图鉴/零件解锁状态变化。
- `TTD.Message.Battle.State.Changed`：战斗状态变化。
- `TTD.Message.Battle.Progress.Changed`：战斗进度和剩余权重变化。
- `TTD.Message.Battle.Currency.Changed`：关卡内货币变化。
- `TTD.Message.Battle.Inventory.Changed`：战斗背包变化。
- `TTD.Message.Battle.Building.Placed`：建筑成功放置。
- `TTD.Message.Battle.Ended`：战斗结束。

## 当前使用点

- `UTTDResearchSubsystem` 在入队、完成、领取、重置、离线进度补算后广播研究所消息。
- `UTTDResearchLabWidget` 监听队列变化与图鉴变化，收到消息后刷新当前页签。
- `UTTDBattleWorldSubsystem` 在战斗状态、进度、货币、背包、建筑放置和结束时广播战斗消息。
- 自动化测试验证 `UGameplayMessageSubsystem` 能注册监听并收到 `FTTDResearchQueueChangedMessage`。

# 全局对象池设计

## 目标

通过 `UTTDObjectPoolWorldSubsystem` 提供每个 `UWorld` 内的全局对象池入口，统一管理敌人、建筑、投射物以及后续高频 UObject 的创建和释放。对象池按 `Class` 作为 key，不使用 GameplayTag 作为业务获取 key。

## 配置入口

对象池配置继续使用 `UTTDDeveloperSettings.GameplayTagDataTables`：

| GameplayTag | Row Struct | 说明 |
| --- | --- | --- |
| `TTD.DataTable.ObjectPool.Definitions` | `FTTDObjectPoolDefinition` | 每个池化 Class 的容量和扩容规则 |

`FTTDObjectPoolDefinition` 字段：

- `ObjectClass`：要池化的 UObject / Actor class。
- `InitialSize`：初始预热数量。
- `MaxSize`：最大池容量，`0` 表示该配置类没有可创建容量。
- `ExpandBy`：容量不足时每次扩容数量，内部至少按 `1` 处理。
- `bPrewarmOnWorldBeginPlay`：是否在 `World BeginPlay` 时按 `InitialSize` 预热。

## 池化接口

只有实现 `UTTDPooledObjectInterface` 的 class 可以进入对象池或被临时创建。接口事件同时支持 C++ 和蓝图：

- `OnPoolCreated(UObject* PoolOwner)`：对象被池创建时触发一次。
- `OnAcquireFromPool(const FTransform& SpawnTransform, UObject* RequestContext)`：业务获取对象时触发。
- `OnReleaseToPool()`：业务释放对象时触发。

Actor 获取时会应用 Transform、显示、启用碰撞和 Tick；回收时会隐藏、禁用碰撞和 Tick。普通 UObject 忽略 Transform 的空间效果，但仍会收到接口事件和 `RequestContext`。

## 运行规则

- 已配置 Class：优先从池中取对象；池为空时按 `ExpandBy` 扩容，直到 `MaxSize`。
- 已配置但达到 `MaxSize`：获取失败，返回 `nullptr` 并记录日志。
- 未配置 Class：如果实现池化接口，会临时创建并返回，`bFromPool=false`，不计入池统计；释放时调用 `OnReleaseToPool` 后销毁 Actor 或交给 GC。
- 未实现接口的 Class：拒绝创建，返回 `nullptr`。
- `ReleaseObject` 对已配置 Class 会校验对象是否属于 ActiveObjects，避免外部对象错误回收到池里。

## 业务 API

- `AcquireObject`：C++ / Blueprint 获取 UObject 或 Actor。
- `AcquireActor`：C++ / Blueprint 获取 Actor，并应用 Transform。
- `ReleaseObject`：释放池化对象或临时对象。
- `PrewarmPool`：手动预热已配置 Class。
- `GetPoolStats`：查询指定 Class 的 `AvailableCount`、`ActiveCount`、`TotalPooledCount`、`MaxSize` 和配置状态。
- C++ 侧提供模板封装 `Acquire<T>()` 和 `Release<T>()`。

# 战斗首版闭环设计

## 目标

实现首版可玩的“守护城堡 + 边界刷怪 + 建筑防守 + 玩具盒获取零件 + 高低差建造格”闭环。该版本优先保证运行链路、配置入口、测试覆盖和失败显式化，暂不做完整关卡选择、正式地图切换和复杂寻路。

## 数据表

战斗配置同样通过 `UTTDDeveloperSettings + GameplayTag -> DataTable` 管理：

| GameplayTag | Row Struct | 说明 |
| --- | --- | --- |
| `TTD.DataTable.Battle.Levels` | `FTTDBattleLevelDefinition` | 关卡 ID、城堡配置、初始图纸/玩具盒/零件/货币、波次列表 |
| `TTD.DataTable.Battle.Waves` | `FTTDWaveDefinition` | 波次延迟、敌人条目、数量、生成间隔、SpawnGroup |
| `TTD.DataTable.Battle.Enemies` | `FTTDEnemyDefinition` | 敌人 Class、血量、移速、攻击、进度权重、货币掉落 |
| `TTD.DataTable.Battle.Buildings` | `FTTDBuildingDefinition` | 建筑 Class、所需图纸、零件消耗、攻击模式、基础属性、投射物 Class |
| `TTD.DataTable.Battle.ToyBoxRewards` | `FTTDToyBoxRewardDefinition` | 玩具盒购买价格、抽取次数、零件权重和数量区间 |
| `TTD.DataTable.Battle.HeightEffects` | `FTTDBattleHeightEffectDefinition` | 高度效果器 ID 和属性修改器列表 |

`UTTDDeveloperSettings.DefaultBattleLevelId` 决定主菜单“关卡挑战”启动的默认关卡。Row 中的 ID 字段为空时，当前加载逻辑会用 RowName 补齐对应 ID。

## 运行时中心

`UTTDBattleWorldSubsystem` 负责：

- 启动和结束战斗。
- 加载战斗 DataTable。
- 扫描场景中已摆放的 `ATTDBattleSpawnPointActor` 和 `ATTDBuildSlotActor`。
- 生成城堡、构建波次 pending spawn 队列、刷怪和计算进度。
- 管理关卡内货币、图纸权限、玩具盒数量和零件数量。
- 校验并执行建造。
- 注册战斗目标，供敌人寻找最近目标。
- 应用高度效果器到建筑运行时属性。
- 广播战斗 GameplayMessage。

首版敌人移动策略是直线追踪最近我方目标。目标集合包含城堡和已建造建筑。胜利条件是所有 pending spawn 清空且场上敌人清空；失败条件是城堡血量归零。

## 进度公式

每个敌人定义带 `ProgressWeight`。关卡总权重在构建刷怪队列时累计：

```text
Progress = 1 - (RemainingUnspawnedWeight + AliveEnemyWeight) / TotalEnemyWeight
```

`RemainingUnspawnedWeight` 表示尚未成功生成的敌人权重，`AliveEnemyWeight` 表示已生成且存活的敌人权重。敌人死亡时扣除存活权重并增加货币掉落。

## 建造与高度效果器

建造格使用场景预摆的 `ATTDBuildSlotActor`。格子包含：

- `SlotId`：格子标识。
- `GridCoordinate`：格子坐标数据。
- `HeightLevel`：高度层级数据。
- `HeightEffectId`：要应用的高度效果器。
- 占用状态。

高度不作为建造限制，也不会固定影响射程。建筑放置时，系统读取 `HeightEffectId` 对应的 `FTTDBattleHeightEffectDefinition`，把其中声明的 `FTTDBattleAttributeModifier` 应用到建筑 `BaseStats`。支持属性：

- `AttackDamage`
- `AttackRange`
- `AttackInterval`
- `MaxHealth`
- `ProjectileSpeed`

支持操作：

- `Add`
- `Multiply`
- `Override`

没有配置高度效果器时，建筑使用基础属性。

## 建筑攻击和投射物

建筑支持两种攻击模式：

- `InstantDamage`：按攻击间隔寻找范围内最近敌人并直接伤害。
- `Projectile`：按攻击间隔生成 `ATTDBattleProjectileActor`，投射物飞向目标并命中伤害。

敌人、建筑、投射物都按池化对象设计。战斗系统优先通过 `UTTDObjectPoolWorldSubsystem` 获取对象；未配置池但实现池化接口时会临时创建；已配置但达到 `MaxSize` 时获取失败。

# P0 隐患修复设计

## 启动前校验

`UTTDBattleWorldSubsystem` 新增带失败原因的启动接口：

- `StartDefaultBattle(FText& OutFailureReason)`
- `StartBattle(FName LevelId, FText& OutFailureReason)`
- Blueprint 包装：`StartDefaultBattleWithReason`、`StartBattleWithReason`

旧的无 reason 版本保留为包装。新的启动流程是：

1. 加载战斗 DataTable。
2. 扫描当前 World 中的 SpawnPoint 和 BuildSlot。
3. 校验关卡、波次、敌人、场景承载和投射物配置。
4. 校验通过后才 `CleanupBattle` 并启动新战斗。

校验失败时不会生成城堡，不会清理当前战斗，不会进入 HUD 假运行状态。`ATTDMenuPlayerController::ShowBattleLevel` 会把失败原因保存给主菜单 UI 显示。

启动校验范围：

- 默认关卡 ID 非空且存在于 `TTD.DataTable.Battle.Levels`。
- 关卡引用的 Wave 存在。
- Wave 引用的 Enemy 存在。
- Enemy Class 为空时允许使用默认 `ATTDBattleEnemyActor`；非空时必须是实现池化接口的 `ATTDBattleEnemyActor` 派生类。
- 每个有数量的 Wave Enemy Entry 都必须能找到对应 SpawnPoint；空 `SpawnGroup` 允许任意 SpawnPoint，但场景至少要有一个 SpawnPoint。
- 场景至少有一个 BuildSlot。
- Castle Class 如果配置，必须派生自 `ATTDBattleCastleActor`。
- Projectile 建筑如果配置 ProjectileClass，必须是实现池化接口的 `ATTDBattleProjectileActor` 派生类；为空时使用默认投射物。

## ConfigurationError

新增 `ETTDBattleState::ConfigurationError` 表示战斗系统配置或运行承载错误，不等同于玩家失败。

`SpawnDueEnemies` 现在先尝试 `SpawnEnemy`，只有生成成功后才：

- 从 `PendingSpawns` 移除该条目。
- 扣除 `RemainingUnspawnedWeight`。
- 广播进度变化。

如果运行中生成失败，例如对象池已配置但 `MaxSize` 为 0 或容量耗尽，系统会：

1. 冻结当前 `GetProgress()`。
2. 进入 `ConfigurationError`。
3. 释放 active projectiles、enemies、buildings。
4. 广播 State / Progress / Ended。

这样可以避免“敌人没生成但权重被扣掉”，进而误判胜利。

## 投射物生命周期

`UTTDBattleWorldSubsystem` 新增：

- `ActiveProjectiles`
- `BattleInstanceId`
- `RegisterProjectile`
- `UnregisterProjectile`
- `IsBattleInstanceCurrent`
- `GetActiveProjectileCount`
- `GetBattleInstanceId`

每次成功启动战斗都会递增 `BattleInstanceId`。建筑发射投射物时，把当前实例 ID 传给 `ATTDBattleProjectileActor::InitializeProjectile` 并注册到 Subsystem。投射物 Tick 时如果发现：

- Subsystem 不存在。
- 当前战斗不是 Running。
- 自己保存的 `BattleInstanceId` 与当前战斗不一致。
- 目标无效。

就会释放自身。战斗结束或清理时，Subsystem 会统一释放所有 `ActiveProjectiles`，再释放敌人和建筑。对象释放优先走 `UTTDObjectPoolWorldSubsystem::ReleaseObject`，只有 Pool 不可用时才 fallback `Destroy()`。

## 验证

- 场景缺少 BuildSlot 时启动失败。
- Wave 引用缺失 Enemy 时启动失败。
